# No-Vibe Planning（决策对齐门禁）

> 一个 Claude Code 技能：**在每个决策都对齐之前，拒绝开工。**

[English](./README.md) | 简体中文

市面上给编码 agent 的"行为准则"大多是*提醒*——"想清楚再写"、"不确定就问"。提醒容易点头、也容易跳过。**No-Vibe Planning 是协议，不是提醒**：它逐个决策分支盘问你的方案，把每条结论钉进决策账本，并把"没对齐"当成写代码的硬性叫停。

## 问题

[Andrej Karpathy 谈 LLM 写代码的通病：](https://x.com/karpathy/status/2015883857489522876)

> "模型替你做出错误假设、然后不加核对就一路跑下去。它们**不管理自己的困惑、不寻求澄清、不暴露不一致、不摆出取舍、该反对时不反对**。"

跟模型说"请寻求澄清"并不能可靠地修好这件事——就像你对自己说"我会先对齐"也挡不住自己滑进代码。问题不在缺好意图，而在缺一个**强制装置**。

## 思路

No-Vibe Planning 把*"我们对齐了吗"*——这件最容易自欺的事——变成一条你被迫走完的循环：

```
先把结构拆开
  → 铺开决策树（每个叉，开篇全摆出来）
  → 逐分支盘问（一次一个，不接受含糊答案）
  → 每条结论钉进决策账本（DECISIONS.md）
  → 每个分支都对齐 + 用户确认前不退出
  → 把对齐好的账本交棒给 设计 → 实现 → 验收
```

四个想法在干活：

| 想法 | 它杀死什么 |
|---|---|
| **一次只钉一个分支** | "一口气甩 20 个澄清问题"——淹没用户、一个都没钉死。手术刀，不是问卷。 |
| **不接受含糊** | "应该没问题"/"到时候再说"被当成对齐记下来。追到具体、可证伪为止。 |
| **压力测试，而非收集** | 老实把一个错误前提记下来。前提有问题，当场指出——对齐是双向的，不是说服。 |
| **不对齐不开工** | 决策树还在一半靠脑补，模型已经跑去实现了。账本必须先清空。 |

## 它和"就问几个澄清问题"差在哪

问问题是第一步。No-Vibe Planning 补上让对齐*粘得住*的三样东西：

1. **一本扛得过上下文丢失的决策账本。** 每条钉死的决策落进 `DECISIONS.md`——决策点、一致结论、*为什么*、一条**可证伪的验收断言**（"60 秒内第 10 次请求后返回 429"，而不是"跑通就行"）。聊天上下文是易失的，文件不会。
2. **一道硬门禁。** 只要还有分支没对齐，就不开工。这是全部意义所在——不是一条你能合理化绕过的建议。
3. **一条交棒链。** 对齐不是散场。对齐好的账本流进 **决策 → 设计 → 实现 → 验收**，每一环用决策号引用上一环，于是下游没有任何块挂在没对齐的决策上（也没有任何决策悄悄漏了实现）。

## 安装

**方式 A —— Claude Code 插件（推荐）**

```
/plugin marketplace add wangjibin555/no-vibe-planning
/plugin install no-vibe-planning@no-vibe-planning
```

然后对任何方案、spec、架构说一句 **"开始对账"** 或 **"开始质询我"** 即可。

**方式 B —— 塞进项目的 `CLAUDE.md`**

```bash
curl https://raw.githubusercontent.com/wangjibin555/no-vibe-planning/main/CLAUDE.md >> CLAUDE.md
```

## 怎么用

- 说 **"开始对账"**、**"开始质询我"**、"挑战我的方案"——立即触发。
- 或者直接开始**共同设计** spec / 架构 / 技术选型——它会自己介入。
- 一次回答一个分支。预期会被反驳。预期听到"你这个前提可能就是错的"。
- 账本清空、你确认达成共识时它才停——然后交棒给 spec → plan → verify。

它**故意不会**在纯执行任务（"改个错别字"、"跑测试"）或你已对齐的事情上触发。你说"别问了，直接做"，它就停——但会把仍未对齐的分支列为已知风险，而不是假装全齐了。

## 怎么知道它在起作用

- 那些你本来要到*实现之后*才会发现的决策，现在*实现之前*就浮出来了。
- "应该没问题"不再是一个可接受的答案。
- 聊天上下文没了之后，你伸手去够的是 `DECISIONS.md`。
- "等等，我不是这个意思"式的返工变少了。
